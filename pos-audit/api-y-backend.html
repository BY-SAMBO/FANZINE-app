API y Backend

La "cocina" del sistema: como se procesan los pedidos, como se comunica con Fudo, y que pasa cuando algo sale mal.

Que hace el backend?

Si el POS es la registradora que usa el cajero, el backend es la cocina del sistema. Recibe los pedidos desde la pantalla, los procesa, los envia a Fudo (el cerebro de facturacion del restaurante), y devuelve la confirmacion.

Analogia: Cuando el cajero completa un pedido y toca "Cobrar", es como si pasara la comanda a la cocina. La cocina (backend) prepara todo, lo registra en la caja fiscal (Fudo), y devuelve el ticket confirmado. Si algo falla en la cocina, el cajero deberia recibir un aviso claro.

El sistema tiene 4 puertas de entrada (endpoints) por donde llegan las solicitudes. Cada una maneja un tipo de operacion diferente.

Los 4 endpoints (puertas de entrada)

POST /api/pos/sale — Crear venta

El endpoint principal. Recibe el pedido completo (productos, toppings, metodo de pago) y crea la venta en Fudo. Es el que se usa cada vez que el cajero cobra un pedido presencial.

Frecuencia de uso: Cada venta presencial (el mas usado)

POST /api/pos/sale/close — Cerrar comanda

Cierra una venta que se creo pero aun no se pago. Util cuando el cliente pide primero y paga despues. Registra el pago y cierra la comanda en Fudo.

Frecuencia de uso: Ocasional (solo para pagos diferidos)

POST /api/pos/modifiers/sync — Sincronizar toppings

Actualiza la lista de toppings disponibles. Los toppings se guardan en un cache local para que no haya que consultar a Fudo cada vez. Este endpoint refresca ese cache.

Frecuencia de uso: Manual (cuando cambian los toppings en Fudo)

POST /api/pos/remote/order — Pedido delivery

Similar al de crear venta, pero para pedidos de delivery a bares. Incluye informacion adicional del destino de entrega.

Frecuencia de uso: Cada pedido de delivery

Como funciona la conexion con Fudo

Fudo es el sistema de facturacion del restaurante. Es donde se registran todas las ventas oficiales, se controla inventario y se generan reportes. Nuestro POS necesita hablar con Fudo para cada venta.

Analogia: Fudo es como el cerebro administrativo del restaurante. Nuestro POS es la mano que toma los pedidos, pero necesita decirle al cerebro lo que vendio para que todo cuadre.

Pasos para crear una venta en Fudo

1

Autenticacion

El sistema se identifica ante Fudo con sus credenciales. Fudo devuelve un "pase" temporal (token) que expira.

2

Crear la venta

Se crea un registro de venta vacio en Fudo, como abrir una nueva comanda en blanco.

3

Agregar productos

Se agregan los productos uno por uno a la comanda. Cada producto tiene su ID de Fudo, cantidad y precio.

4

Agregar toppings

Para cada producto que tiene extras, se agregan los toppings como "subitems" del producto principal.

5

Registrar pago

Se registra como se pago (efectivo, tarjeta, etc.) y el monto total.

6

Cerrar la venta

Se cierra la comanda en Fudo. A partir de aqui la venta es oficial y aparece en los reportes.

Importante   Si alguno de estos pasos falla (por ejemplo, el paso 4), la venta queda abierta en Fudo pero incompleta. No hay un mecanismo automatico para reintentar o limpiar ventas incompletas.

Que pasa cuando algo falla?

Cuando ocurre un error, el sistema deberia responder de forma clara. Esta tabla muestra las situaciones de error y como se manejan actualmente:

Situacion Que pasa hoy Que deberia pasar
Fudo no responde Error generico en pantalla Reintentar automaticamente 2 veces, luego mostrar opcion de reintento manual
Token expirado Se renueva automaticamente Funciona correctamente
Producto sin ID de Fudo Error 500 sin detalle Identificar el producto problematico y mostrarlo al cajero
Venta parcialmente creada Se pierde (queda abierta en Fudo) Intentar cerrar/cancelar la venta parcial
Impresora desconectada Ticket se pierde silenciosamente Guardar en cola y reintentar cuando la impresora este disponible
Internet se cae Error generico Guardar pedido localmente y enviar cuando vuelva la conexion

Seguridad

Estos son los hallazgos de seguridad. Aunque el POS opera en una red interna, es importante corregirlos:

!

Credenciales guardadas sin encriptar

Las credenciales de Fudo (usuario y contrasena del sistema) estan en un archivo de configuracion sin cifrar. Si alguien accede al servidor, puede leerlas directamente. Es como guardar la llave de la caja fuerte debajo del tapete.

!

Tablas sin politicas de acceso (RLS)

Las tablas de la base de datos no tienen restricciones de acceso. Cualquier persona con la clave publica del proyecto podria leer o modificar datos. Es como dejar los libros contables abiertos sobre la mesa.

!

Cache que nunca se refresca automaticamente

La informacion de toppings y metodos de pago se guarda localmente y nunca se actualiza automaticamente. Si se agregan opciones nuevas en Fudo, no aparecen hasta que alguien manualmente ejecute la sincronizacion.

Tablas de base de datos

El sistema usa tablas en la base de datos para guardar informacion. Algunas de estas tablas fueron creadas manualmente y no estan documentadas en "migraciones" (instrucciones automaticas para recrear la base de datos).

Tabla Que guarda En migraciones?
products Catalogo de productos del restaurante Si
categories Categorias de los productos Si
pos_modifier_cache Cache de toppings sincronizados de Fudo No
pos_sales_log Historial de ventas realizadas No
delivery_config Configuracion de delivery por producto Si

Critico   Las tablas pos_modifier_cache y pos_sales_log no tienen migraciones. Si se necesita reinstalar el sistema o crear un ambiente de pruebas, estas tablas habra que crearlas manualmente, con riesgo de errores.
