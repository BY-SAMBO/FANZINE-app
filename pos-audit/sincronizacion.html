Sincronizacion en Tiempo Real

Como se comunican la pantalla del cajero y la del cliente, y que pasa cuando se pierde la conexion.

Por que necesitan comunicarse dos pantallas?

El POS de FANZINE usa dos pantallas que trabajan juntas: la del cajero y la del cliente. Funcionan como un equipo: el cajero selecciona el producto y el cliente elige los extras (toppings).

Analogia: Imagine dos personas jugando cartas juntas. Una reparte (el cajero elige producto) y la otra juega (el cliente elige toppings). Para que el juego funcione, ambas necesitan ver las mismas cartas al mismo tiempo. Si una de las dos deja de escuchar, el juego se traba.

Esta comunicacion en tiempo real es lo que permite que el cliente vea inmediatamente lo que el cajero agrega, y que el cajero vea los toppings que el cliente selecciona.

Dos formas de comunicacion

El sistema usa dos metodos para enviar mensajes entre pantallas:

BroadcastChannel

"Como gritar en la misma sala"

Funciona cuando ambas pantallas estan abiertas en el mismo navegador (dos pestanas o ventanas en la misma computadora).

  • Instantaneo (menos de 1ms)
  • No necesita internet
  • Solo funciona en el mismo equipo
  • Se pierde si cierran la pestana

Supabase Realtime

"Como walkie-talkies"

Funciona cuando las pantallas estan en dispositivos diferentes (ej: computadora del cajero y tablet del cliente).

  • Funciona entre dispositivos
  • Necesita internet
  • Latencia de 50-200ms
  • Puede perder mensajes si se cae la conexion

El sistema intenta usar ambos metodos al mismo tiempo para mayor compatibilidad. Pero esto crea un problema: si ambos funcionan, el mensaje puede llegar duplicado.

Los 5 tipos de mensaje

Las pantallas se comunican usando 5 tipos de mensajes. Aqui estan explicados como una conversacion:

Caja → Cliente
1. "Nuevo producto"

El cajero agrego un producto al pedido. La pantalla del cliente debe mostrar las opciones de toppings para ese producto.

Caja → Cliente
2. "Pedido actualizado"

El pedido cambio (se quito un producto, se cambio cantidad, etc.). La pantalla del cliente se actualiza.

Cliente → Caja
3. "Toppings seleccionados"

El cliente selecciono sus extras. La pantalla del cajero los agrega al producto correspondiente.

Caja → Cliente
4. "Pedido cobrado"

El cajero completo el cobro. La pantalla del cliente muestra un mensaje de agradecimiento.

Ambos
5. "Pedido limpiado"

Se limpio el pedido. Ambas pantallas vuelven a su estado inicial.

Que pasa cuando se pierde la conexion?

Actualmente, el sistema no maneja bien las desconexiones. Estos son los escenarios problematicos:

Escenario 1: El cliente pierde internet por 30 segundos

Mientras esta desconectado:

  1. El cajero agrega 3 productos al pedido
  2. Los mensajes se envian pero la pantalla del cliente no los recibe
  3. Cuando vuelve la conexion, la pantalla del cliente sigue mostrando el pedido viejo
  4. El cajero no sabe que el cliente esta desincronizado

Resultado: Las pantallas muestran informacion diferente. El cliente puede estar eligiendo toppings para un pedido que ya no existe.

Escenario 2: El cajero recarga la pagina

Cuando el cajero recarga:

  1. El pedido en memoria se pierde completamente
  2. La pantalla del cliente sigue mostrando el pedido anterior
  3. No hay sincronizacion automatica para que ambas pantallas se pongan de acuerdo

Resultado: El cliente ve un pedido "fantasma" que ya no existe en la caja.

Escenario 3: Se cae el wifi del restaurante

Sin wifi:

  1. Supabase Realtime deja de funcionar
  2. BroadcastChannel sigue funcionando si estan en el mismo equipo
  3. Pero si estan en dispositivos separados, no hay comunicacion
  4. Cuando vuelve el wifi, no se reconectan automaticamente

Resultado: Hay que recargar ambas paginas manualmente cuando vuelve la conexion.

Sistema de codigo de sesion

Como funciona

Para que las pantallas sepan que pertenecen al mismo "puesto", usan un codigo de sesion. Actualmente se usa un canal fijo llamado pos:caja:main.

Esto significa que solo puede haber un puesto de caja activo al mismo tiempo. Si alguien abre una segunda caja, ambas compartiran el mismo canal y se interferiran.

Riesgos del sistema actual

  • Un solo canal: No soporta multiples puestos de caja simultaneos
  • Sin autenticacion: Cualquier pestana que se conecte al canal puede enviar mensajes
  • Sin heartbeat: No hay manera de saber si la otra pantalla sigue activa
  • Sin numeracion: Si un mensaje se pierde, no hay forma de detectarlo
