Estado y Flujo de Datos

Como el sistema recuerda la informacion de los pedidos y como viaja esa informacion desde que el cajero selecciona un producto hasta que llega la factura.

Que es el "estado" del sistema?

Piense en el estado como la memoria inmediata del sistema. Asi como un mesero recuerda mentalmente lo que va pidiendo la mesa mientras toma la orden, el sistema POS recuerda en su "estado" los productos que se han agregado, los toppings seleccionados, y el metodo de pago.

Analogia: Imagine que el mesero usa una libreta para la orden. El "estado" es lo que esta escrito en esa libreta en este momento. Si el mesero pierde la libreta (se recarga la pagina), se pierde la orden.

El sistema tiene dos libretas separadas (dos "stores"): una para la caja presencial y otra para los pedidos de delivery. Veremos por que esto es un problema.

En cada paso, la informacion se va acumulando en el estado. Al final del proceso, todo se envia a Fudo (el sistema de facturacion del restaurante) y se imprime el ticket para la cocina.

Las dos "libretas" del sistema

Actualmente el sistema mantiene dos copias del codigo que maneja los pedidos. Es como si el restaurante tuviera dos menus impresos que hay que mantener actualizados por separado.

pos-store.ts (Caja)

Maneja los pedidos presenciales. Es la version principal.

  • Agrega/quita productos
  • Controla toppings
  • Procesa cobros
  • Envia a Fudo

remote-pos-store.ts (Delivery)

Maneja los pedidos para bares/delivery. Tiene el 70% del mismo codigo.

  • Misma logica de productos
  • Mismos toppings
  • Mismo proceso de cobro
  • Agrega campo de destino
El problema: Si se encuentra un error en como se calculan los toppings y se corrige en la caja, hay que acordarse de corregirlo tambien en delivery. Si se olvida, delivery sigue con el error. Es como corregir un precio en un menu pero olvidar corregirlo en la carta del delivery.

Ademas, existe un tercer archivo llamado pos-live-store.ts que es una copia vieja que ya no se usa. Su presencia solo genera confusion.

Como funcionan los toppings

Los toppings (extras como queso, tocineta, etc.) tienen un flujo especial porque involucran dos pantallas: la del cajero y la del cliente.

El proceso paso a paso:

  1. El cajero agrega un producto (ej: hamburguesa) al pedido
  2. El sistema envia un mensaje a la pantalla del cliente: "Elige tus toppings"
  3. El cliente ve las opciones y selecciona (ej: queso extra, sin cebolla)
  4. Las selecciones viajan de vuelta y se guardan junto al producto
  5. El cajero continua con el siguiente producto o cobra
Analogia: Es como si el mesero le mostrara la carta de extras al cliente, el cliente senalara lo que quiere, y el mesero lo anotara junto al plato correspondiente en su libreta.

Problemas encontrados

Critico   El "Producto Fantasma"

Los toppings pueden terminar asignados al producto equivocado.

Ejemplo real:
1. Cajero agrega hamburguesa → Cliente ve pantalla de toppings
2. Mientras el cliente elige, el cajero rapidamente agrega papas fritas
3. El cliente selecciona "queso extra"
4. El queso extra queda asignado a las papas fritas, no a la hamburguesa

Causa: El sistema siempre asigna toppings al ultimo producto agregado, sin importar cual producto se estaba personalizando.

Critico   El "Gemelo Olvidado"

Existe un archivo duplicado que no hace nada pero confunde.

Que pasa:
El archivo pos-live-store.ts es una version vieja del store principal. No se usa en ningun lado, pero si alguien necesita hacer un cambio, podria editarlo por error pensando que es el correcto, y su cambio nunca tendria efecto.

Solucion: Simplemente eliminarlo.

Importante   El "Precio sin Actualizar"

Si se cambia un precio en Fudo, el pedido puede usar el precio viejo.

Ejemplo:
A las 10am, los productos se cargan del servidor con precio de hamburguesa = $25,000.
A las 2pm, alguien sube el precio en Fudo a $28,000.
Si la pagina del POS sigue abierta sin recargar, seguira usando $25,000.

Riesgo: Se factura a precio viejo durante todo el turno si no se recarga la pagina.

Recomendaciones

1. Unificar las dos "libretas" en una sola

Crear un solo store compartido entre caja y delivery. Los cambios se hacen una vez y aplican para ambos. El codigo de delivery solo agrega la parte del destino.

2. Bloquear asignacion de toppings al producto correcto

Cuando el cliente esta eligiendo toppings para un producto, el sistema debe "anclar" esa seleccion a ese producto especifico, sin importar que el cajero agregue otros productos mientras tanto.

3. Eliminar el archivo duplicado

Borrar pos-live-store.ts inmediatamente. No se usa y solo genera confusion.

4. Refrescar precios periodicamente

  • Revisar precios del servidor cada 30 minutos
  • Mostrar una alerta si algun precio cambio desde la ultima carga
  • Validar precios contra el servidor antes de finalizar la venta
