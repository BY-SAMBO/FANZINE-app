Las 4 pantallas del POS

La Caja

/pos/caja

La pantalla principal del cajero. Aqui se arma el pedido: se eligen categorias, se agregan productos, se seleccionan toppings y se cobra. Es como la registradora moderna del restaurante.

Pantalla del Cliente

/pos/cliente

Lo que ve el cliente en la segunda pantalla. Cuando no hay pedido muestra la marca, cuando hay productos muestra las opciones de toppings, y al final muestra un "gracias".

Pedidos a Bares

/pos/bares

Igual que la caja, pero para pedidos de delivery a bares. Tiene un campo extra para seleccionar el bar de destino. El 70% del codigo es identico a la caja.

Seguimiento

/pos/seguimiento

Muestra el estado de los pedidos de delivery en tiempo real. Permite ver que pedidos estan pendientes, en camino o entregados.

Problemas de experiencia de usuario

Estos son los problemas que afectan la experiencia diaria del cajero y del cliente:

No hay confirmacion antes de borrar el pedido

Si el cajero toca accidentalmente el boton de limpiar, se borra todo el pedido sin preguntar. Es como si el mesero pudiera tirar la comanda a la basura con un solo toque accidental. Deberia preguntar "Esta seguro?" antes de borrar.

Botones muy pequenos para pantalla tactil

Algunos botones tienen menos de 44x44 pixeles, que es el tamano minimo recomendado para pantallas tactiles. En el ritmo rapido de una cocina, es facil tocar el boton equivocado. Los botones de toppings son especialmente pequenos.

No funciona en celular ni tablet

Las pantallas fueron disenadas para monitores grandes (1920px). En una tablet de 10" o un celular, los elementos se sobreponen, el texto se corta, y algunos botones quedan fuera de la pantalla. Si algun dia se necesita tomar un pedido desde una tablet, no sera posible.

Metodo de pago habilitado cuando no deberia

El selector de metodo de pago esta activo desde el inicio, incluso cuando no hay productos en el pedido. Esto puede confundir porque sugiere que se puede cobrar un pedido vacio.

Sin retroalimentacion al usuario

Muchas acciones pasan en silencio. Cuando se agrega un producto no hay animacion ni sonido. Cuando se envia a Fudo no hay indicador de carga. Cuando falla algo, no siempre se muestra un error claro. El usuario no sabe si el sistema esta procesando o si se trabo.

Codigo duplicado: el mismo trabajo 3 veces

El componente de toppings (que permite elegir extras como queso, tocineta, etc.) esta escrito 3 veces de forma casi identica en el sistema:

Caja
Toppings en la pantalla del cajero
~120 lineas de codigo
=
Cliente
Toppings en la pantalla del cliente
~115 lineas de codigo
=
Delivery (Bares)
Toppings en pedidos a bares
~120 lineas de codigo

Que implica esto? Si se necesita agregar un nuevo topping, hay que editar 3 archivos. Si se corrige un error en uno y se olvida en otro, los toppings funcionan diferente en cada pantalla. La solucion es crear un solo componente compartido.

Accesibilidad

La accesibilidad asegura que el sistema sea usable para todas las personas. Estos son los puntos evaluados:

X
Contraste de texto insuficiente en algunos botones (texto gris claro sobre fondo oscuro)
X
Botones sin etiquetas de texto accesible (los lectores de pantalla no saben que hacen)
X
No se puede navegar usando solo el teclado (Tab y Enter)
!
Tamano de fuente fijo (no se adapta a preferencias del usuario)
!
Areas tactiles menores a 44x44px en varios botones
X
Sin indicadores de foco visible al navegar con teclado

Por que importa: Aunque el POS lo usa principalmente el cajero, mejorar la accesibilidad beneficia a todos. Botones mas grandes y mejor contraste reducen errores incluso para personas sin discapacidades, especialmente en ambientes oscuros como el del restaurante.
